# 待跟踪问题

本文档用于持续跟踪后续需要处理的问题，排序规则如下：

1. 简单的先做，复杂的后做
2. bug 修复优先于功能需求
3. 复杂度相同，重要程度高的先做
4. 明确的先做，不明确的后做

后续处理方式：按本文档顺序逐项提取、处理、补充结论。

状态约定：

- 未开始：尚未分析或处理
- 处理中：正在分析、实现或验证
- 已完成：已确认完成
- 暂缓：当前不处理，但保留跟踪

### 1. 模型调用出错无提示
状态：已完成
说明：需要像 Web 一样显式展示模型错误、等待重试或类似状态，避免用户误以为只是卡住。
参考：`docs/superpowers/specs/2026-05-11-session-model-error-surfacing-note.md`

- Android 端当前并非完全拿不到错误/重试数据

- 当前 Android 把 `retry` 粗暴映射成 `BUSY`，导致 message / attempt / next 被丢弃，用户只会看到普通忙碌，容易误判为卡死
- 已实现 assistant 终态错误卡片展示
- 已实现 `session.status` 的 retry 细节解析与会话页显式展示
- 已改为 `SSE` 实时更新 + 轮询兜底，符合移动端不能只依赖 `SSE` 的约束

### 2. 工具调用被中止/取消之后，似乎还会显示执行中转动图标
状态：已完成
说明：中止后 UI 状态未正确收敛，容易误导用户认为工具仍在执行。
结论：

- 已确认服务端在中止清理后，会把未完成工具收敛为 `error`，错误文案为 `Tool execution aborted`
- Android 会话页当前走 `SessionMessageRenderer` 直接读取 assistant `content` 中的 tool 状态；若增量同步未把工具从 `running` 改成 `error`，就会持续显示执行中 spinner
- 根因确认在 Android `EventReplayer`：处理 `session.next.step.failed` 时只结束了 assistant，自身没有把仍处于 `pending/running` 的 tool 一并收敛为 aborted error
- 已补最小修复：`step.failed` 时会把未结束工具统一转成 `error`，并写入 `Tool execution aborted` 与完成时间
- 已在安卓端实际对话中复测，中止后显示为 `aborted`，spinner 不再残留

下一步：若后续再遇到残留 spinner 的具体场景，再补充更细的分类复现与验证。

### 3. compact 消息不会展示，导致用户不知道进展
状态：已完成
说明：需要让 compact 过程在消息流中可见。
结论：

- 已确认 `compact` 事件没有在同步阶段丢失；`EventReplayer` 会把 `session.next.compaction.started/ended` 重放为顶层 `type="compaction"` 的消息，并写入本地数据库
- 根因在渲染层：`SessionMessageRenderer` 顶层只处理 `user` / `assistant` / `synthetic`，导致已同步到本地的顶层 `compaction` 消息被直接跳过
- 已补最小修复：为顶层 `type="compaction"` 增加可见渲染，显示 `▸ compaction`
- 已补回归测试：`feature/session` 新增 `SessionMessageRendererCompactionTest`，验证顶层 `compaction` 消息会显示
- 已验证：`./gradlew.bat "-Pandroid.testInstrumentationRunnerArguments.class=com.openmate.feature.session.component.SessionMessageRendererCompactionTest" :feature:session:connectedDebugAndroidTest --no-daemon` 通过；`./gradlew.bat :feature:session:testDebugUnitTest --tests com.openmate.feature.session.component.SessionMessageRendererTest --no-daemon` 通过

下一步：若后续希望展示更细的 compact 进度或摘要，再单独增强文案与状态展示。

- 已补测试：`SessionDetailViewModelTest.loadSession_restoresRunningAnchorFromIncompleteMessageAge` 与 `SubtaskDetailScreenRunningAnchorTest`
- 已由用户在手机上验证：退出会话重新进入后，未结束消息计时不会再从 0 开始

下一步：暂无；若后续再遇到特殊消息类型计时恢复异常，再按具体类型补充回归测试。

### 5. 中止功能调用后，有时 UI 反馈上看还会继续计时
状态：已完成
说明：需要调查计时是否没有正确停止，或只是展示状态未同步。
结论：

- 已确认不是服务端一直未结束，而是 Android 会话页在 `abort` 成功后仍要等后续同步刷新，期间 `currentBusyStart` / `runningAnchors` 还保留，UI 会短暂继续计时
- 根因在 `SessionDetailViewModel.abort()`：原实现只调用 `abortSession()` 后立刻 `refresh()`，没有对当前内存消息窗口做本地立即收敛
- 已补最小修复：`abort` 成功后先将当前窗口内未结束的 assistant 本地标记为已完成，同时把仍处于 `pending/running` 的 tool 状态本地收敛为 `Tool execution aborted`，并立即清空 `currentBusyStart` 与 `runningAnchors`
- 已补回归测试：`SessionDetailViewModelTest.abort_stopsBusyTimerImmediatelyBeforeSyncCatchesUp`
- 已验证：`./gradlew.bat :feature:session:testDebugUnitTest --no-daemon` 通过；`./gradlew.bat :app:assembleDebug --no-daemon` 通过

下一步：若后续发现中止后远端又回写不同终态文案，再按服务端最终状态补充更细的本地收敛策略。
