### 2. 工具调用被中止/取消之后，似乎还会显示执行中转动图标
状态：已完成
说明：中止后 UI 状态未正确收敛，容易误导用户认为工具仍在执行。
结论：

- 已确认服务端在中止清理后，会把未完成工具收敛为 `error`，错误文案为 `Tool execution aborted`

- 已补最小修复：`step.failed` 时会把未结束工具统一转成 `error`，并写入 `Tool execution aborted` 与完成时间
- 已在安卓端实际对话中复测，中止后显示为 `aborted`，spinner 不再残留

下一步：若后续再遇到残留 spinner 的具体场景，再补充更细的分类复现与验证。

结论：

- 已确认不是服务端一直未结束，而是 Android 会话页在 `abort` 成功后仍要等后续同步刷新，期间 `currentBusyStart` / `runningAnchors` 还保留，UI 会短暂继续计时

- 已补回归测试：`SessionDetailViewModelTest.abort_stopsBusyTimerImmediatelyBeforeSyncCatchesUp`

- 用户后续真机反馈：已完成子任务可点击，运行中的子任务仍缺少末尾箭头且不可点击

- 已补最小修复：`extractToolItems()` 现在会把 `structured.sessionId/sessionID` 合并回 `metadata`，保持现有 `TaskToolLine` 导航判定逻辑不变
- 已补回归测试：`SessionMessageRendererTest.extractToolItems_promotesStructuredSessionIdForRunningTaskTool`

- **最终根因确认**：通过拉取真机 DB 与 API 数据对比发现：
  - 运行中 task 的 `sessionId` 不在 `state.structured`（空 `{}`），也不在 `state.metadata`（不存在）

- 已补最小修复：`EventReplayer.message.part.updated` 现在也处理 `type="tool"` 的 part 更新，将 `metadata`、`title`、`structured`、`content` 合并回 cached assistant 中对应 tool 的 state
- 已补回归测试：`EventReplayerTest.replay_messagePartUpdated_mergesToolMetadataForRunningTask` 和 `replay_messagePartUpdated_keepsExistingToolStateFieldsWhenMerging`
