# OpenMate 服务端总体设计

## 1. 产品定位

OpenMate 服务端是连接移动端与 Bridge Agent 的平台层，核心价值：

- **稳定可靠的中继**：为 App 和 Bridge 提供点对点通讯通道，突破局域网限制
- **平台化管理**：用户体系、实例管理、管理后台

## 2. 演进阶段

| 阶段 | 目标 | 核心能力 |
|------|------|----------|
| **Phase 1** | 最小可用中继 | Rust 转发网关 + HTTP/SSE 隧道 + 局域网/远程双模式配对 |
| **Phase 2** | 用户体系 | .NET 业务服务 + GitHub OAuth + JWT 签发 + 实例管理 |
| **Phase 3** | P2P 优化 | STUN 信令 + P2P 直连尝试 + TURN 后备，降低服务器带宽成本 |

### 职责划分

| 组件 | 核心职责 | 不做的事 |
|------|----------|----------|
| Rust 网关 | HTTP/SSE 接入、WS 隧道管理、消息中继、心跳 | 不碰业务逻辑、不连数据库 |
| .NET 服务 | 用户体系、认证、实例管理、Web UI | 不处理实时转发 |
| Bridge | SSE→隧道上行、隧道→本地 HTTP 代理 | 新增 WS 上行模块，现有功能不变 |
| Android | 局域网优先，网关中转后备 | 增加 ConnectionManager fallback 逻辑 |

### 服务间通信

- Rust 网关启动时通过 HTTP 调 .NET 获取 JWT 验证公钥，缓存本地
- 运行时网关独立运行，不依赖 .NET 可用性
- 部署形态：Rust 网关可水平扩展（前面放 LB），.NET 服务单实例起步

## 4. 技术栈

| 组件 | 语言 | 框架 | 数据库 | 说明 |
|------|------|------|--------|------|
| Rust 转发网关 | Rust | Axum + Tokio | 无 | 高性能无状态中继 |
| .NET 业务服务 | C# | ASP.NET Core | MongoDB | 用户体系 + 业务逻辑 |

## 5. Phase 1 范围

Phase 1 只实现 Rust 转发网关，.NET 服务在 Phase 2 开始。

Phase 1 的认证方案：复用 Bridge 现有的 HMAC-SHA256 token 机制，扩展 token 结构以支持网关路由。JWT 公钥验证在 Phase 2 由 .NET 签发后接入。

详见 [Phase 1 详细设计](./phase1-relay-gateway.md)。
