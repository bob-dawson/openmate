# 删除旧代码清单

## 目标

- 清理 Android 端旧 `Message/Part` 架构残留，减少新会话详情页开发和日志排查时的干扰。
- 采用多轮推进，优先删除几乎确定是老代码、无人引用或仅被老测试引用的内容。
- 明确保留当前新链路：凡是名称包含 `SessionMessage` 的代码，本轮一律不纳入删除范围。

## 已确认策略

- `SessionMessage` 相关代码全部视为当前新消息结构，不纳入删除清单。
- `PartRenderer + Part.kt` 这一整套先保留，放到第二轮再评估。
- `fetchFullMessage / session_message_full_content` 完整消息缓存链保留，不删除。
- 增量同步相关新文档可对照更新后替换旧同步设计文档。
- `android/AGENTS.md` 中旧消息结构描述后续要改成基于新的消息结构。

## 第一轮

### 删除代码

- `android/core/domain/src/main/java/com/openmate/core/domain/repository/MessageRepository.kt`
  - 旧消息仓库接口
  - 当前仅被 `MessageRepositoryImpl` 和 `DataModule` 旧绑定引用

- `android/core/data/src/main/java/com/openmate/core/data/repository/MessageRepositoryImpl.kt`
  - 旧仓库实现
  - 当前为空壳实现：`observeMessages()` 直接返回空列表

- `android/feature/session/src/main/java/com/openmate/feature/session/component/MessageSearchPanel.kt`
  - 旧搜索面板
  - 当前会话详情页实际使用的是 `SessionMessageSearchPanel`

### 删除测试

- `android/core/domain/src/test/java/com/openmate/core/domain/model/MessageTest.kt`
  - 仅覆盖旧 `Message` / `MessageRole`

### 同步清理引用

- `android/core/data/src/main/java/com/openmate/core/data/DataModule.kt`
  - 删除 `bindMessageRepository(impl: MessageRepositoryImpl): MessageRepository`

### 第一轮执行备注

- `FileAttachment` 虽然原来定义在旧 `MessageRepository.kt`，但当前新会话详情上传链路仍在使用。
- 第一轮删除时不应连带删除该类型；已改为独立保留，不再依附旧仓库接口。

## 第二轮候选

### 旧模型与旧渲染链

- `android/core/domain/src/main/java/com/openmate/core/domain/model/Message.kt`
- `android/core/domain/src/main/java/com/openmate/core/domain/model/Part.kt`
- `android/core/domain/src/main/java/com/openmate/core/domain/model/MessageRole.kt`
- `android/feature/session/src/main/java/com/openmate/feature/session/component/PartRenderer.kt`
- `android/core/domain/src/test/java/com/openmate/core/domain/model/PartTest.kt`

说明：

- 这批仍和 `MessageDto.kt`、`OpencodeApiClient.getMessages()`、旧 DTO 映射链有耦合。
- 用户已明确：`PartRenderer + Part.kt` 先保留到第二轮，不在第一轮处理。

### 第二轮盘点结论（已确认）

- `Message.kt` 与 `MessageRole.kt`
  - 当前已无 UI / ViewModel / Repository 主链直接引用。
  - 仅剩 `android/core/network/src/main/java/com/openmate/core/network/dto/MessageDto.kt` 仍在使用。
  - 结论：若后续改掉 `MessageDto.kt -> Message/Part` 映射，这两个文件可以一并删除。

- `Part.kt`
  - 当前仍被 `MessageDto.kt`、`PartRenderer.kt`、`PartTest.kt`、`RealDataParsingTest.kt` 使用。
  - 结论：仍是旧链核心模型，暂不单独删除。

- `PartRenderer.kt`
  - 当前仍有真实代码引用，不能视为无人引用。
  - 结论：需等新消息渲染路径完全覆盖后再删。

- `MessageDto.kt`
  - 当前 `OpencodeApiClient.getMessages()` 仍返回 `MessageWithPartsDto`。
  - 结论：这是第二轮及以后要处理的关键耦合点，删除 `Message.kt` / `MessageRole.kt` / `Part.kt` 前需要先重构这里。

- `RealDataParsingTest.kt`
  - 当前主要覆盖 `MessageWithPartsDto` / `PartDto` 解析。
  - 结论：等 `MessageDto.kt` 重构后再一起删或重写。

### 下一轮最小切口（已确认）

- 当前 `OpencodeApiClient.getMessages()` 的真实调用方只有一处：
  - `android/core/data/src/main/java/com/openmate/core/data/repository/SessionRepositoryImpl.kt`
  - 用途仅是判断：是否存在 `role == "assistant" && time.completed == null` 的最新消息

- 这意味着 `getMessages()` 当前并不需要旧 `MessageWithPartsDto -> Message/Part` 全量映射能力。

- 最小替代方向：
  - 在 `MessageDto.kt` 中保留纯网络层 DTO，但移除 `toDomain(): Message` 与 `toDomain(): Part`
  - 或进一步把 `getMessages()` 改成只返回一个更轻量的消息列表 DTO，只保留：
    - `info.id`
    - `info.role`
    - `info.time.created`
    - `info.time.completed`
    - `parts` 可直接保留原始 JSON 或继续保留 DTO，但不再映射到旧 domain `Message/Part`

- 删除顺序建议：
  1. 先改 `OpencodeApiClient.getMessages()` 返回结构，去掉对 domain `Message` / `MessageRole` / `Part` 的依赖
  2. 再删除 `Message.kt` 与 `MessageRole.kt`
  3. 再评估 `Part.kt`、`PartRenderer.kt`、`RealDataParsingTest.kt`

- 风险判断：
  - 这一轮不应直接动 `Part.kt` / `PartRenderer.kt`
  - 应优先把 `MessageDto.kt` 从“网络 DTO + 旧 domain 映射”收缩为“仅网络 DTO”
