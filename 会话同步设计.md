# 会话消息同步设计

## 背景

OpenMate Android 客户端需要从 opencode 服务端同步会话消息到本地 Room 数据库。消息一旦完成（`completedAt != null`）就不会再变更，这为增量同步提供了基础。

## API 分页机制

`GET /session/:sessionID/message?limit=N&before={cursor}`

- 无 `before`：返回**最新 N 条消息**，按 `createdAt ASC` 排列
- 有 `before`：返回比 cursor **更旧**的 N 条消息
- 响应 header `X-Next-Cursor`：下一页（更旧方向）的 cursor，base64 编码的 `{id, time}` JSON
- 无更多数据时，响应不包含 `X-Next-Cursor` header
- API **没有 `after` cursor**，只能向更旧方向翻页

### 消息特征

- user 消息：只有 `time.created`，没有 `time.completed`
- assistant 消息：有 `time.created` 和 `time.completed`（可为 null，表示正在 streaming）
- 已完成的消息（`completed != null`）不会变更
- 未完成的消息（`completed == null`）后续会更新

## 核心概念

### 同步锚点（Sync Anchor）

每个会话维护一个**同步锚点**：该会话中最后一条 `completedAt != null` 的 assistant 消息的 ID。

- 含义：该消息及比它更旧的所有已完成消息，已经可靠同步到本地
- 锚点之前的消息不需要重新拉取
- 锚点之后的消息（包括未完成的）每次同步都需要覆盖

### 存储位置

在 `SessionEntity` 中新增字段 `syncAnchor: String?`，存储该会话的同步锚点消息 ID。

## 同步流程

### 首次加载（无锚点）

```
1. GET /message?limit=80  →  获取最新 80 条
2. upsertAll 写入 DB
3. 检查返回数据中最后一条 completed 的 assistant 消息，记为锚点
4. 如果 X-Next-Cursor 存在（还有更旧数据），按需加载（用户上滑触发）
5. 保存锚点到 SessionEntity.syncAnchor
```

### 增量同步（有锚点，Poll 触发）

增量同步应使用较小的 limit，因为通常只有少量新消息。如果未找到锚点，再翻页继续查找。

```
1. GET /message?limit=5  →  获取最新 5 条
2. upsertAll 写入 DB
3. 检查返回数据中是否包含锚点消息
   - 包含锚点：同步完成，更新锚点为当前最新已完成 assistant 消息
   - 不包含锚点：说明新消息超过 5 条，需要继续往前翻页
4. 往前翻页：GET /message?limit=10&before={cursor}（limit 翻倍）
5. upsertAll，继续检查是否包含锚点
6. 重复直到找到锚点或没有更多数据，每次翻页 limit 翻倍（5→10→20→40→80，上限 80）
7. 更新锚点为本次同步中最新一条已完成 assistant 消息
```

**limit 策略**：
- 增量同步首页：`limit=5`
- 每次翻页 limit 翻倍：5 → 10 → 20 → 40 → 80（上限 80）
- 首次加载：`limit=80`（一次性加载较多历史消息）
- 最多 5 次请求可覆盖 155 条新消息，常见场景（1-2 条新消息）只需 1 次请求

### 向前翻页（用户上滑加载历史）

```
1. 使用上次保存的 cursor（X-Next-Cursor）请求更旧的消息
2. upsertAll 写入 DB
3. 不更新锚点（这些是更旧的消息，不影响锚点）
```

### 同步中断恢复

- 同步过程中断网/崩溃，锚点未更新
- 下次同步时，仍从旧锚点开始匹配，之前已 upsert 的消息被 REPLACE 覆盖（幂等）
- 直到找到锚点后，更新为新锚点
- **不会产生数据空洞**

## 关键规则

1. **锚点只认已完成的 assistant 消息**：`role == ASSISTANT && completedAt != null`
2. **user 消息不参与锚点判断**：user 消息没有 `completed` 字段
3. **upsertAll 是幂等的**：重复写入相同 ID 的消息不会出错，REPLACE 策略覆盖
4. **未完成的消息不设锚点**：`completedAt == null` 的 assistant 消息下次仍会被重新拉取
5. **没有锚点 = 全量同步**：首次加载或锚点丢失时，翻到没有更多数据为止

## 会话忙碌状态判断

- 不再依赖 `/session/status` API
- 基于 DB 中该会话是否存在 `role == ASSISTANT && completedAt == null` 的消息
- `observeMessages` Flow 实时监测，状态变化自动反映到 UI

## 旧数据处理

开发阶段直接清库重来，无需处理旧版 bug 产生的脏数据。

## 时序图

```
客户端                          服务端
  |                               |
  |-- GET /message?limit=80 ----->|  首次加载
  |<-- 80条消息 + X-Next-Cursor --|
  |                               |
  |  upsertAll → DB               |
  |  保存锚点                      |
  |                               |
  |-- (15s 后 poll) -------------|  增量同步
  |-- GET /message?limit=10 ----->|  小 limit，通常足够
  |<-- 最新10条 -------------------|
  |                               |
  |  包含锚点 → 同步完成           |
  |  更新 syncAnchor              |
  |                               |
  |-- (假设新消息很多) -----------|  增量同步（翻页场景）
  |-- GET /message?limit=10 ----->|
  |<-- 10条（不含锚点）+ cursor --|
  |                               |
  |-- GET /message?limit=20      -->|  翻页，更大 limit
  |   &before={cursor}            |
  |<-- 更旧的20条 + cursor -------|
  |                               |
  |  找到锚点 → 同步完成           |
  |  更新 syncAnchor              |
```

## 与 SSE 的关系

同步完成后，可对该会话建立 SSE 连接，实时接收 `message.part.delta`、`message.updated` 等事件，直接更新 DB。SSE 是同步的补充，不是替代：

- 同步：保证数据完整性（拉取全量/增量消息）
- SSE：保证实时性（streaming 文本、状态变更）

当前阶段 SSE 未启用，poll 作为唯一的实时更新手段。
