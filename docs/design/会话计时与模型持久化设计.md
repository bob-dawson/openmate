# 会话计时与模型持久化设计

## 1. 计时设计

### 1.1 核心原则

- 统一用 epoch 毫秒，以 PC 时间为准（PC/手机都有自动对时，无需额外处理时差）
- busy 开始时间 = 最后一条 `role=user && roundMark=true` 的消息的 timeCreated
- 所有时间计算统一使用 PC 时间（消息的 timeCreated），不使用手机本地时间

### 1.2 交互轮次标记：roundMark

SessionMessageEntity 新增 `roundMark: Boolean` 字段，每条消息都写。

**user 消息的 roundMark**（只看前一条消息）：

| 前一条消息 | roundMark |
|-----------|-----------|
| 无（首条消息） | true |
| assistant, roundMark=true | true |
| assistant, roundMark=false | false |
| user, roundMark=任意 | false |

简化规则：**`roundMark = 无前一条 || (前一条.role == "assistant" && 前一条.roundMark == true)`**

**assistant 消息的 roundMark**（本条即可判断）：

| finish | error | 含义 | roundMark |
|--------|-------|------|-----------|
| null | null | 进行中 | false |
| `tool-calls` | null | 中间步骤，loop 继续 | false |
| `unknown` | null | 不确定，loop 继续 | false |
| `stop` | null | 自然结束 | true |
| `length` | null | 达到上限 | true |
| null | `MessageAbortedError` | 用户中止 | true |
| null | `APIError` 等 | 出错 | true |

简化规则：**`roundMark = finish ∈ ["stop", "length"] || error != null`**

**写入时机**：
- EventReplayer 处理 `message.updated` 事件时计算并写入
- 首次同步拉取消息时，`toEntity()` 中计算
- SSE `message.part.delta` 流式更新时不改变 roundMark（finish 尚未到达）
- 写入 user 消息时，可缓存最后一条消息的 `role + roundMark`，避免 DB 查询

### 1.3 数据模型

SessionEntity 保留：
- `totalDuration: Long?` — 累计已完成时长（毫秒），累加而非覆盖

SessionEntity 删除（逻辑中不再使用，schema 保留）：
- `startedAt`
- `phoneStartedAt`

SessionMessageEntity 新增：
- `roundMark: Boolean` — 交互轮次边界标记

ViewModel 内存：
- `currentBusyStart: Long?` — 当前 busy 段的开始时间
- `lastMessageMeta: Pair<String, Boolean>?` — 缓存最后一条消息的 (role, roundMark)，写入时免查询

### 1.4 计时显示公式

```
busy 时：totalDuration + max(0, System.currentTimeMillis() - currentBusyStart)
idle 时：totalDuration
从未活跃过：不显示
```

### 1.5 状态转换

```
observeMessages 触发：
  扫描消息列表，判断是否有未完成 assistant（role=assistant && roundMark=false）：
  
  有未完成 assistant（busy）：
    1. 找最后一条 role=user && roundMark=true 的消息，取 timeCreated 作为 currentBusyStart
    2. 如果 currentBusyStart > now → currentBusyStart = now（防负数）
    3. 实时显示 = totalDuration + max(0, now - currentBusyStart)
  
  无未完成 assistant（idle）：
    如果之前是 busy（wasBusy=true）：
      1. increment = max(0, now - (currentBusyStart ?: now))
      2. totalDuration = (totalDuration ?: 0) + increment
      3. 写 DB
      4. currentBusyStart = null
    否则：无操作
```

```
进入会话（loadSession）：
  1. 从 DB 读取 totalDuration
  2. observeMessages 会自动计算 currentBusyStart
```

```
退出会话：
  如果当前 busy：
    累加 increment 到 totalDuration，写 DB
    （确保退出时持久化）
```

### 1.6 场景覆盖

| 场景 | currentBusyStart 来源 | 显示 |
|------|----------------------|------|
| 手机在线发消息 | turn-start user.timeCreated | 递增 |
| 连续发 2 条消息（busy 中） | 不变，第二条 user roundMark=false | 整段递增 |
| TUI 发起，手机在线 | 同上，PC timeCreated | 递增 |
| 用户中止 | — | busy→idle，累加到 totalDuration |
| 退出再进入（busy） | 从 roundMark=true 的 user 消息重算 | 继续递增 |
| 退出再进入（idle） | null | totalDuration |

### 1.7 roundMark 的额外价值

- 支持按交互轮次查看消息（未来功能）
- 可直接查询某轮交互的所有消息：`WHERE seq >= turnStartSeq AND seq <= turnEndSeq`
- 统计交互轮次：`SELECT COUNT(*) WHERE roundMark=true AND role='user'`

### 1.8 关键改动

1. SessionMessageEntity 新增 roundMark 字段
2. 删除 _phoneStartedAt / _sessionStartedAt
3. 新增 _currentBusyStart（内存，从 roundMark 计算）
4. totalDuration 改为累加
5. busy→idle 不清空任何持久化字段
6. sendMessage 不再设 phoneStartedAt
7. 退出会话时如果 busy，持久化当前增量
8. 缓存 lastMessageMeta，写入时免查询

## 2. 模型持久化

### 2.1 数据模型

SessionEntity 新增：
- `modelProviderID: String?`
- `modelID: String?`
- `modelName: String?`

### 2.2 流程

进入会话：DB 有值 → 立即显示 → 异步 providers 仅刷新列表
用户选模型 / assistant 消息含模型 → 写 DB

## 3. DB Migration

版本 15 → 16：

```sql
ALTER TABLE SessionEntity ADD COLUMN modelProviderID TEXT;
ALTER TABLE SessionEntity ADD COLUMN modelID TEXT;
ALTER TABLE SessionEntity ADD COLUMN modelName TEXT;
ALTER TABLE SessionMessageEntity ADD COLUMN roundMark INTEGER NOT NULL DEFAULT 1;
```

`startedAt` / `phoneStartedAt` 字段保留在 schema（已有数据），逻辑中不再使用。

移除 fallbackToDestructiveMigration，改用正式 migration。

### 旧数据 roundMark 处理

Migration 默认值 `roundMark = 1`（true）。旧数据一律视为轮次边界，这是最安全的默认值：
- 旧 assistant 消息 roundMark=true → 视为已结束，不会误判为 busy
- 旧 user 消息 roundMark=true → 视为轮次起点
- 新 user 消息发送时，前一条旧 assistant roundMark=true → 自然标记为新轮次开始
- 无需补算逻辑

## 4. 实现步骤

### Step 1: DB Migration
1. SessionMessageEntity 新增 roundMark 字段
2. SessionEntity 新增 modelProviderID/modelID/modelName
3. Session/SessionEntity 映射增加模型字段
4. SessionMessageDto.toEntity() 中计算 roundMark
5. SessionDao: 新增 updateModel()，updateDuration 改为累加
6. AppDatabase version 16 + Migration(15,16)
7. 移除 fallbackToDestructiveMigration

### Step 2: 计时逻辑重构
1. ViewModel: 删除 _phoneStartedAt/_sessionStartedAt，新增 _currentBusyStart
2. observeMessages: 用 roundMark 判断，找最后一条 role=user && roundMark=true 的消息计算 currentBusyStart
3. busy→idle 累加 totalDuration
4. loadSession: 只从 DB 恢复 totalDuration
5. sendMessage: 移除 phoneStartedAt 设置
6. 实时计时 UI: totalDuration + max(0, now - currentBusyStart)
7. 退出会话时持久化未保存的增量

### Step 3: 模型持久化
1. loadSession 从 DB 恢复 selectedModel
2. selectModel 写 DB
3. observeMessages 从 assistant 消息提取模型信息写 DB
4. resolveDefaultModel 仅当 DB 无值时调 API

### Step 4: SessionRepositoryImpl merge
1. merge 时保留 modelProviderID/modelID/modelName
2. getSession/getSessions upsert 逻辑更新

## 5. 调研备注

### opencode Runner 行为（已验证）

- `ensureRunning` 在 Running 状态时，新 prompt 的 `runLoop` work 被 await，不会中断当前 run
- 但 `runLoop` 内部是 `while(true)` 循环，每次迭代重新读取所有消息，自然吸收新 user 消息
- 第二条 user 消息不会被丢弃，而是被同一个 run 的下一轮迭代处理
- 整个过程是一个连续的 busy 段，没有 busy→idle→busy 的切换

### 用户中止时的消息状态（已验证）

- `finish = null`，`error = MessageAbortedError`，`completedAt` 有值
- `finish` 为 `tool-calls` 的中间步骤也有 `completedAt`，所以不能用 `completedAt` 判断是否最终结束
- roundMark 统一处理：`error != null` 也算轮次结束

### opencode v2 消息无轮次概念（已确认）

- User 消息只有 `time.created`，无 turn/round 标记
- Assistant 消息有 `finish` + `error` + `time.completed`，无轮次标记
- `step-start`/`step-finish` 是 assistant 内部多步循环，不是交互轮次
- roundMark 为我们自行实现的字段，不存在与 opencode 重复
