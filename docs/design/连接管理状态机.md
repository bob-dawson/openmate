# 连接管理状态机

## 状态列表

| 状态 | 携带数据 | 说明 |
|---|---|---|
| Idle | profile? | 未连接，无活跃会话 |
| ProbingNetwork | profile, attempt | 查网络+缓存，决定走哪条路 |
| WaitingForNetwork | profile, attempt | 等网络恢复 |
| ProbingDirect | profile, attempt | TCP 探测直连 Bridge |
| ProbingGateway | profile, attempt | HTTP 探测网关在线 |
| ConnectingCached | profile, route, attempt | 缓存命中的 SSE 连接 |
| ConnectingFresh | profile, route, attempt | 探测后的 SSE 连接 |
| Connected | profile, route, attempt | SSE 已连接 |
| Recovering | profile, attempt | 退避等待，准备重新探测 |
| Failed | profile, reason | 所有路由不可达 |
| NeedsRepair | profile | Bridge 需要配对授权 |

## 路由类型

| Route | 数据 | 说明 |
|---|---|---|
| Direct | address, port | 直连 LAN/Tailscale |
| Gateway | instanceId | 经云网关中继 |

| Effect | 数据 | 执行内容 |
|---|---|---|
| CheckNetwork | profile, attempt | 查网络+缓存 → 发 CacheXxx / NetworkIsNone |
| ProbeDirect | — | TCP 探测直连 → 发 ProbeOk/ProbeFail/BridgeNotBridge/BridgeNeedsRepair |
| ProbeGateway | — | HTTP 探测网关 → 发 ProbeOk/ProbeFail |
| StartSse | route | 按 route 启动 SSE 连接 |
| StopSse | — | 断开 SSE |
| StartBackoff | delayMs | 开始退避定时器 |
| StopBackoff | — | 取消退避定时器 |
| RefreshSessions | — | 刷新会话列表 |
| UpdateLastConnectedAt | profileId | 更新 profile.lastConnectedAt |
| StartDirectCheckLoop | — | 每 30s 探测直连是否恢复 |
| StopDirectCheckLoop | — | 停止直连探测循环 |
| RestartDirectCheckLoop | — | 重启直连探测循环 |
| WriteCacheDirect | profileId | DataStore 写 direct 缓存 |
| WriteCacheGateway | profileId | DataStore 写 gateway 缓存 |
| ClearCache | profileId | DataStore 清除缓存 |

## kstatemachine 使用规则

1. **所有 `targetState` 必须声明式设置**，禁止在 `onTriggered` 内动态赋值（会导致 `onEntry` 不触发）
2. **禁止同一状态内对同一事件类型定义多个 transition + guard**（kstatemachine 会报 `Multiple transitions match` 崩溃）
3. **需要条件分支时，拆成不同状态**（如 `ConnectingCached` vs `ConnectingFresh`）
4. **`onTriggered` 只设 `_state.value`，副作用归 `onEntry/onExit`**

## 退避算法

```
delay = min(1000ms × 2^min(attempt, 4), 30000ms)
attempt 0 → 1s
attempt 1 → 2s
attempt 2 → 4s
attempt 3 → 8s
attempt 4 → 16s
attempt 5+ → 30s（上限）
```

## 缓存路由逻辑

1. `ProbingNetwork` onEntry 调 `CheckNetwork` → `EffectExecutor.checkNetwork()`
   - 无网络 → `NetworkIsNone` → `WaitingForNetwork`
   - 有网络 + 缓存=direct → `CacheDirectOk` → `ConnectingCached`
   - 有网络 + 缓存=gateway → `CacheGatewayOk` → `ConnectingCached`
   - 有网络 + 无缓存 → `CacheNone` → `ProbingDirect`
2. `Connected` onEntry 写缓存（Direct 写 direct，Gateway 写 gateway）
3. `ConnectingCached` SseFailed → `ClearCache` → `ProbingGateway`（缓存失效，跳过退避直接探测网关）
4. `ConnectingFresh` SseFailed → `Recovering`（正常退避重试）

## App 前后台处理

### 方案：进后台主动断开，进前台从 Idle 重连

- **AppForegroundMonitor** 使用 `ProcessLifecycleOwner`（进程级前后台检测），替代旧的 Activity 计数方式
- **进后台**（`ProcessLifecycleOwner.onStop`）：发送 `AppBackgrounded`
  - `Connected` 收到 → `StopSse` → `Idle(profile)`（保留 profile）
  - 其他状态不处理 `AppBackgrounded`
- **进前台**（`ProcessLifecycleOwner.onStart`）：发送 `AppForegrounded`
  - `Idle(profile!=null)` → `ConnectionManager.connect(profile)` → 立即重连
  - `WaitingForNetwork` → `ProbingNetwork`
  - `Failed` → `ProbingNetwork`

| 状态 | AppBackgrounded | AppForegrounded |
|---|---|---|
| Idle | ❌ | ✅（ConnectionManager 层，profile!=null → connect） |
| ProbingNetwork | ❌ | ❌ |
| WaitingForNetwork | ❌ | ✅ → ProbingNetwork |
| ProbingDirect | ❌ | ❌ |
| ProbingGateway | ❌ | ❌ |
| ConnectingCached | ❌ | ❌ |
| ConnectingFresh | ❌ | ❌ |
| Connected | ✅ → Idle | ❌ |
| Recovering | ❌ | ❌ |
| Failed | ❌ | ✅ → ProbingNetwork |
| NeedsRepair | ❌ | ❌ |

**优势**：进后台时主动断开 SSE，进前台时状态一定是 Idle（无"假连接"），直接重连不经过退避。
