# 代码搜索索引优化方案

## 问题背景

OpenMate 项目（Android Kotlin + Rust Bridge + Rust Server）代码搜索依赖 grep/glob/read，在几千文件的多模块项目中效率低。测试了 semble 等 AI 代码搜索工具，发现：

- **semble**：宣称 250ms 索引，实测 Windows 完全卡死，WSL 下 33 秒（CPU 4.5s + I/O 28.5s），每次全量重建不可用
- **CodeSage**（Rust）：不支持 Kotlin
- **codedb**（Zig）：Windows 需编译，Kotlin 仅轻量 outline，无语义搜索

semble 的核心问题：每次搜索都全量扫描+读取+embedding+建索引，无法利用已有的索引结果。

## 方案思路

基于 semble 的 potion-code-16M 静态 embedding 模型（CPU 可用、无需 GPU），加上持久化增量索引：

### 1. 持久化索引

用 SQLite 存储：
- 文件元信息：`file_path, file_size, mtime, language`
- 代码块：`chunk_id, file_path, start_line, end_line, content`
- 向量：`chunk_id, embedding`（numpy 数组序列化或专用向量表）
- BM25 token：`chunk_id, tokens_json`

### 2. 增量更新（关键）

**不能用 content hash**——读取文件算 hash 就失去了避免 I/O 的意义。

用 `stat()` 获取 `mtime + file_size` 做 O(1) 的变更检测：

- 启动时遍历文件目录，`stat()` 每个文件取 mtime+size
- 与 SQLite 中记录的 mtime+size 对比
- **新增文件**：chunk → embed → BM25 tokenize → 写入索引
- **变更文件**（mtime 或 size 不同）：删除旧 chunk/embedding/token → 重新处理
- **删除文件**：从索引中移除
- **未变更文件**：跳过，直接用缓存数据

### 3. BM25 增量重建

BM25 索引（bm25s）目前不支持增量添加。方案：
- 保留 per-chunk 的 token 列表在 SQLite
- 查询时从 SQLite 加载全部 token → 在内存重建 BM25（几秒，比全量扫描快得多）
- 或：维护一个增量 BM25 结构（需自行实现或找支持增量的 BM25 库）

### 4. 向量索引增量重建

Vicinity（semble 使用的向量库）也不支持增量。方案：
- 保留 per-chunk 的 embedding 在 SQLite
- 查询时从 SQLite 加载全部 embedding → 在内存重建向量索引
- 预建索引模式下，这部分可以预加载到内存常驻

### 5. 模型加载

potion-code-16M 静态模型约 16MB，首次下载后缓存本地，加载时间 <1 秒。MCP 常驻模式下只加载一次。

### 6. 搜索流程

```
用户查询 → 加载模型（MCP 模式已缓存）
         → 从 SQLite 读 embedding → 内存建向量索引
         → 从 SQLite 读 tokens → 内存建 BM25
         → hybrid search (RRF 融合 + reranking)
         → 返回结果
```

冷启动（首次 MCP 连接）预估：
- stat 遍历 + 增量检测：~1 秒
- 从 SQLite 加载 embedding/token：~2-5 秒（取决于 SQLite 读取速度）
- 内存建索引：~1-2 秒
- 总计：首次约 5-10 秒，后续查询 <100ms

热查询（索引已在内存）：
- 直接查询，~1.5ms（semble benchmark 的数据）

## 实现优先级

1. 先实现 CLI 模式验证可行性（预建索引 + SQLite 持久化 + mtime 增量检测）
2. 确认 potion-code-16M 模型在目标环境可用
3. 再做 MCP 常驻模式
