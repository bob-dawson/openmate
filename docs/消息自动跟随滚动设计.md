# 消息自动跟随滚动设计

## 核心概念

**followBottom**：一个布尔状态，表示"是否应该自动跟随底部"。
- `true`：新消息到来时自动滚动到底部
- `false`：新消息到来时不滚动，保持当前位置

## 场景分析

### 1. 初始消息加载

**期望**：首次进入会话，加载消息后滚动到底部。

**触发**：`prevCount == 0` 且 `messageCount > 0`

**行为**：
- `followBottom` 初始为 `true`
- 延迟 150ms（等布局完成）后滚动到底部
- 这是唯一不需要 `followBottom = true` 也能滚动的场景

### 2. 新消息同步增加（自动跟随）

**期望**：当 `followBottom = true` 且有新消息同步到 DB 时，自动滚动到底部。

**触发**：`messageCount` 变化（Flow 发射新列表）

**行为**：
- `followBottom = true` 且 `canScrollForward = true` → 滚动到底部
- `followBottom = false` → 不滚动

### 3. 手动滚动到非底部

**期望**：用户手动向上滚动，离开底部区域，停止自动跟随。

**难点**：如何区分"用户手指滚动"和"布局变化导致的滚动"（如键盘弹出、消息内容更新导致高度变化）。

**判断方式**：
- 用户主动滚动：`isScrollInProgress = true` 且 `canScrollForward = true`
- 布局变化：`isScrollInProgress = false`（或为 true 但非用户触发的动画）

**问题**：键盘弹出时 Compose 可能报告 `isScrollInProgress = true` 且 `canScrollForward` 短暂为 false（因为列表被压缩），然后变为 true。

**行为**：
- 检测到 `isScrollInProgress = true` 且 `canScrollForward = true` → `followBottom = false`
- 之后的 `messageCount` 变化不触发滚动

### 4. 手动滚回底部（恢复跟随）

**期望**：用户手动滚到底部（`canScrollForward = false`），恢复自动跟随。

**难点**：键盘弹出时 `canScrollForward` 也可能变为 false。

**区分方案**（待定）：
- **方案 A**：`isScrollInProgress = true` 且 `canScrollForward = false` → `followBottom = true`
  - 问题：键盘弹出可能也满足条件
- **方案 B**：只通过"滚到底部"按钮恢复跟随
  - 问题：用户手动滚到底不会恢复，体验差
- **方案 C**：用 `isScrollInProgress` 持续时间区分，短于阈值（如 100ms）的视为布局变化
  - 复杂但更准确
- **方案 D**：检查 `listState.firstVisibleItemIndex` 变化方向，向下滚动到末尾才恢复
  - 可能更可靠

### 5. 消息搜索定位

**期望**：搜索定位到某条消息后，停留在该位置，不被后续操作拉回底部。

**触发**：搜索面板点击某条消息

**行为**：
- 设置 `userNavigating = true` → `followBottom = false`
- `animateScrollToItem(index)` 滚动到目标
- 滚动完成后 `userNavigating = false`
- **关键**：`userNavigating = false` 后，`followBottom` 仍为 `false`，不会自动滚回

**问题**：
- 定位后如果 `animateScrollToItem` 动画过程中短暂到达底部（`!canScrollForward`），可能触发 `followBottom = true`
- 需要在搜索定位动画期间禁止 `followBottom` 变回 `true`

### 6. 输入法弹出/收起

**期望**：输入法弹出或收起时，不改变 `followBottom` 状态。

**难点**：
- 键盘弹出导致 LazyColumn 可视区域缩小，Compose 可能报告：
  - `isScrollInProgress = true`（布局重排触发）
  - `canScrollForward` 短暂变化
- 这不是用户主动滚动，不应改变 `followBottom`

**解决思路**：
- 键盘弹出时 `isScrollInProgress` 的触发与用户滚动不同，但没有 API 区分
- 可能需要监听 `WindowInsets.ime` 变化来标记"键盘动画中"，期间忽略滚动状态变化

## 当前实现的问题

1. `isScrollInProgress` 无法区分用户滚动和布局变化
2. `canScrollForward` 在键盘弹出时不可靠
3. `animateScrollToItem` 动画中可能触发 `followBottom` 状态变化
4. `snapshotFlow` 对组合状态变化的触发时机不确定

## 待评审方案

### 方案：imeInsets 感知 + 用户滚动检测

```
状态变量：
- followBottom: Boolean = true
- autoScrolling: Boolean = false  // 自动滚动动画中
- isKeyboardAnimating: Boolean = false  // 键盘动画中
- userNavigating: Boolean = false  // 搜索定位中

状态转换规则：

1. 初始加载 → followBottom = true, 滚动到底

2. messageCount 变化：
   - followBottom = true && !userNavigating → 滚动到底
   - 否则 → 不滚动

3. isScrollInProgress 变为 true：
   - autoScrolling || isKeyboardAnimating → 忽略
   - canScrollForward = true → followBottom = false（用户向上滚了）
   - canScrollForward = false → followBottom = true（用户滚到底了）

4. userNavigating = true → followBottom = false

5. requestFollow = true → followBottom = true（滚到底按钮）

6. WindowInsets.ime 变化中 → isKeyboardAnimating = true，忽略滚动事件
```

### 关键点：如何检测键盘动画

```kotlin
// 监听 IME insets 变化
val imeInsets = WindowInsets.ime
LaunchedEffect(imeInsets) {
    isKeyboardAnimating = true
    delay(300) // 等键盘动画完成
    isKeyboardAnimating = true = false
}
```

### 关键点：如何防止 animateScrollToItem 误触发

在 `scrollToBottom()` 开始时设 `autoScrolling = true`，结束设 `false`。
在 `autoScrolling = true` 期间，忽略所有 `isScrollInProgress` 事件。

### 关键点：搜索定位保护

搜索定位时设 `userNavigating = true`，此时：
- `messageCount` 变化不触发滚动
- `isScrollInProgress` 事件被忽略（`canScrollForward = false` 不设 `followBottom = true`）
- 动画完成后设 `userNavigating = false`，但 `followBottom` 保持 `false`

## 其他考虑

- `scrollToBottom()` 使用 `animateScrollToItem`，对于大量消息可能较慢
- 如果 `canScrollForward` 在动画中一直为 true，`needScroll` 的 `derivedStateOf` 不会误触发（因为 `followBottom` 已为 true 且正在滚动）
- `prevCount` 用于判断是否是初始加载，`prevCount == 0` 表示首次加载
