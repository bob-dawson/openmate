# Plan: message.updated abort 检测修复

## 问题

用户发消息后 assistant 还没产生就被 abort。`message.updated` 带着旧 assistant（前一轮的）的 completed 到达，但由于 `hadCachedAssistant == true` 保护，abort 检测被跳过 → 新 user message 后面没有 completed assistant → 显示忙碌。

## 根因

当前 `message.updated` handler（line 636）只在 `!hadCachedAssistant` 时运行 abort 检测。但 abort 场景中：
1. 旧 assistant 在 cache 中（由前一轮 step.started 创建），`hadCachedAssistant == true`
2. `message.updated` 更新旧 assistant 的 completed → 正确
3. 但 abort 检测被 `hadCachedAssistant` 守卫跳过 → 新 user message 后没有 assistant

## 关键区分

`message.updated` 的 `info` 中有 `time.created`：
- **Abort**: assistant.timeCreated < 最后一条 user message.timeCreated（旧轮次的 assistant 被关闭）
- **正常多步**: assistant.timeCreated > 对应 user message.timeCreated（当前轮次的 assistant 完成）

## 修改

### EventReplayer.kt — `message.updated` handler

移除 `hadCachedAssistant` 守卫，改用 `info.time.created` 对比 DB 中是否有更新的 user message：

```kotlin
"message.updated" -> {
    val info = props["info"]?.jsonObject ?: return emptyList()
    val role = info["role"]?.jsonPrimitive?.contentOrNull
    if (role != "assistant") return emptyList()
    val time = info["time"]?.jsonObject ?: return emptyList()
    val completed = time["completed"]?.jsonPrimitive?.longOrNull ?: return emptyList()
    val infoCreated = time["created"]?.jsonPrimitive?.longOrNull ?: return emptyList()

    var cached = getCachedAssistant()
    if (cached == null) {
        ensureAssistantCache(sessionId, loader)
        cached = getCachedAssistant()
    }

    if (cached != null) {
        val timeObj = cached.second["time"]?.jsonObject?.toMutableMap() ?: mutableMapOf()
        if (timeObj.containsKey("completed")) return emptyList()
        timeObj["completed"] = JsonPrimitive(completed)
        val updated = cached.second.toMutableMap().apply {
            put("time", JsonObject(timeObj))
            if (!containsKey("finish")) put("finish", JsonPrimitive("error"))
        }
        val merged = JsonObject(updated)
        setCache(cached.first, "assistant", merged, cached.third)
        val changes = mutableListOf(ReplayChange.Update(cached.first, "assistant", merged, timestamp, completedAt = completed))

        // Abort 检测：用 info.time.created（不是 cache 的 timeCreated）检查是否有更新的 user message
        val afterUser = loader(DbLoader.Action.HasNewerUserMessageAfter(sessionId, infoCreated))
        if (afterUser != null) {
            val abortedId = afterUser.id + "_abort"
            if (loader(DbLoader.Action.LoadById(abortedId)) == null) {
                val abortedData = buildJsonObject {
                    put("content", JsonArray(emptyList()))
                    put("time", buildJsonObject {
                        put("created", JsonPrimitive(afterUser.timeCreated))
                        put("completed", JsonPrimitive(completed))
                    })
                    put("finish", JsonPrimitive("error"))
                }
                setCache(abortedId, "assistant", abortedData, afterUser.timeCreated)
                changes += ReplayChange.Insert(SessionMessageEntity(
                    id = abortedId,
                    sessionId = sessionId,
                    type = "assistant",
                    data = abortedData.toString(),
                    timeCreated = afterUser.timeCreated,
                    timeUpdated = completed,
                    roundMark = true,
                    completedAt = completed,
                ))
            }
        }
        return changes
    }
    return emptyList()
}
```

**核心改动**：
1. 从 `info.time.created` 取 `infoCreated`
2. 用 `infoCreated`（不是 cache 的 timeCreated）调 `HasNewerUserMessageAfter`
3. 移除 `hadCachedAssistant` 守卫 — 不管 cache 来源如何，只要 `info.time.created` < 新 user message 的 timeCreated，就插入 abort 占位符

**为什么安全**：正常多步调用中，`message.updated` 的 `info.time.created` 是当前轮次 assistant 的创建时间，总是 > 对应 user message 的 timeCreated，所以 `HasNewerUserMessageAfter` 返回 null，不会触发 abort。
